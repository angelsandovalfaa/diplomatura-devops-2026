Hoy vamos a meternos en un tema que, para cualquier persona que trabaje en desarrollo, infraestructura o automatización, tarde o temprano se vuelve indispensable. Vamos a hablar de Git, de GitHub, de GitLab, y sobre todo de una idea mucho más importante que los nombres de las herramientas... la idea de controlar cambios, trabajar en equipo sin caos, y poder construir software sin miedo a romper todo en el intento.

Porque pensemos esto un segundo... imaginá que estás escribiendo un documento importante con otra persona. Vos corregís una parte, la otra persona cambia otra, después alguien borra algo sin querer, más tarde quieren volver a cómo estaba el archivo ayer, y de pronto nadie sabe cuál era la versión correcta. Bueno, en el desarrollo de software eso pasa todo el tiempo, pero multiplicado por miles de archivos, decenas de personas y cambios constantes. Ahí es donde aparece un sistema de control de versiones.

Un sistema de control de versiones, dicho de manera muy simple, es una herramienta que guarda la historia de un proyecto. No solamente guarda el estado actual, sino también qué cambió, cuándo cambió, quién lo cambió y, algo fundamental, por qué se cambió. Es como si el proyecto tuviera memoria. Y esa memoria no solo sirve para mirar al pasado con nostalgia... sirve para trabajar mejor en el presente y para resolver problemas cuando algo sale mal.

Dentro de ese mundo, Git es el sistema de control de versiones más usado. Y no es casualidad. Git se volvió tan importante porque permite algo muy poderoso: que cada persona tenga una copia completa del repositorio, es decir, una copia completa del proyecto con todo su historial. Eso hace que el trabajo sea mucho más flexible y seguro. No dependés de estar todo el tiempo conectado a un único servidor para poder avanzar. Podés trabajar localmente, guardar cambios, crear ramas, probar ideas y después sincronizar todo cuando corresponda.

Acá aparece una diferencia útil para entender por qué Git marcó tanto la historia del desarrollo. Antes existían sistemas centralizados, donde el repositorio principal vivía en un servidor y todos dependían de ese punto central para casi todo. Git, en cambio, es distribuido. Y cuando decimos distribuido, no estamos usando una palabra elegante porque sí. Queremos decir que el conocimiento del proyecto está repartido. Cada colaborador tiene una copia completa. Es como si cada integrante del equipo tuviera no solo una fotocopia de la última página de un libro, sino el libro entero, con todas sus ediciones anteriores.

Eso trae varias ventajas. Una es la velocidad. Otra es la seguridad. Otra es la facilidad para trabajar en paralelo. Y ahí llegamos a un concepto clave: las ramas. Una rama, o branch, es como una línea paralela de trabajo. Imaginá un árbol. El tronco principal puede representar la versión estable del proyecto. Pero mientras tanto, otras ramas crecen a los costados, cada una explorando una mejora, corrigiendo un error o probando una idea nueva. Lo lindo de esto es que podés trabajar sin tocar directamente la versión principal. Entonces, si algo falla, no rompés todo. Podés experimentar con más tranquilidad.

Y esto cambia muchísimo la forma de trabajar. Porque ya no hace falta que todo el equipo toque el mismo lugar al mismo tiempo. Cada persona puede desarrollar una funcionalidad en su propia rama, hacer pruebas, corregir detalles, y recién después proponer que esos cambios se integren al proyecto principal. Esa capacidad de separar trabajo, ordenar cambios y después unirlos con criterio es una de las razones por las que Git es tan valioso.

Ahora bien, para que Git nos resulte útil en la práctica cotidiana, hay ciertos conceptos y acciones básicas que conviene entender. Primero está la idea del repositorio. Un repositorio es, básicamente, el espacio donde Git guarda el proyecto y todo su historial. Después aparece el working directory, que es el lugar donde estamos editando archivos. Más adelante aparece el staging area, que es como una bandeja intermedia, un espacio donde elegimos qué cambios queremos preparar para guardar en el historial. Y finalmente aparece el commit.

Un commit es una fotografía con contexto. Y esto es importante... no es solo una foto del estado del proyecto, sino una foto acompañada por una explicación. Cuando alguien hace un commit, está diciendo: “estos cambios forman una unidad con sentido, y quiero registrarlos en la historia del proyecto”. Por eso el mensaje del commit importa tanto. Es la pequeña narración que le explica al futuro qué pasó en ese momento.

Pensalo como si estuvieras escribiendo un diario de trabajo. Si cada entrada dijera solamente “cosas”, no serviría de mucho. Pero si cada una dice “corregí el error del login”, “agregué validación al formulario” o “preparé la integración con la base de datos”, entonces la historia se vuelve útil. Git hace justamente eso: convierte cambios dispersos en una secuencia ordenada de decisiones.

Después están las acciones de sincronización. Porque una cosa es trabajar localmente, y otra es compartir ese trabajo con otras personas o con un repositorio remoto. Ahí entran ideas como subir cambios, traer cambios y fusionar historias. Subir cambios significa publicar lo que hiciste para que el repositorio remoto también lo tenga. Traer cambios significa incorporar lo que otras personas hicieron. Y fusionar, o mergear, significa combinar dos líneas de trabajo en una sola.

Y acá es donde aparece una de las situaciones más humanas del desarrollo colaborativo: el conflicto. Un conflicto ocurre cuando dos cambios afectan la misma parte de un archivo y Git no puede decidir solo cuál debe quedar. Lejos de ser una tragedia, eso es una señal de que dos historias tocaron el mismo punto y ahora necesitan una decisión humana. Resolver conflictos no es “arreglar un error del sistema”, es tomar una decisión consciente sobre cómo deben integrarse cambios distintos.

También aparece una herramienta muy útil para la vida real del desarrollador, que es la posibilidad de guardar trabajo temporalmente cuando todavía no querés convertirlo en commit. A veces estás en medio de algo, pero necesitás cambiar de rama o atender otra tarea. En esos casos, Git permite apartar ese trabajo momentáneamente, como si guardaras cosas en una caja con una etiqueta que dice “lo sigo después”. Esa lógica ayuda mucho cuando el trabajo no avanza en una sola línea recta, que es lo normal en casi cualquier proyecto.

Hasta acá hablamos de Git como herramienta. Pero en el trabajo real, Git casi siempre aparece acompañado por plataformas remotas. Y ahí entran GitHub y GitLab. Ambos permiten alojar repositorios, colaborar con otras personas, revisar cambios, registrar issues, organizar trabajo y disparar flujos automáticos. En otras palabras, si Git es el motor del control de versiones, GitHub y GitLab son entornos donde ese motor se vuelve social, visible y colaborativo.

GitHub es, probablemente, la plataforma más popular y conocida. Muchísimas comunidades de código abierto viven ahí. GitLab, por su parte, también es una plataforma muy potente, con un fuerte enfoque en DevOps y con muchas capacidades integradas. Las dos permiten crear repositorios remotos, invitar colaboradores, definir permisos, revisar cambios y sostener flujos de trabajo en equipo.

Lo importante no es entrar en una pelea de marcas, sino entender qué problema resuelven. Pensemos en esto... trabajar solo con Git local sería como escribir un libro excelente, pero dejarlo guardado en un cajón de tu casa. GitHub y GitLab permiten compartirlo, revisarlo, continuarlo con otras personas y darle una vida colectiva. Son el puente entre tu trabajo local y el trabajo del equipo.

Y en ese puente aparecen conceptos muy importantes como fork, pull request y merge request. Un fork es una copia de un repositorio en otra cuenta. Es muy útil, por ejemplo, en proyectos abiertos, donde alguien quiere proponer mejoras sin tocar directamente el repositorio original. Es como sacar una copia de una receta, hacer tus modificaciones en tu cocina y después decir: “miren, encontré una forma mejor de prepararla, ¿les gustaría incorporarla?”.

Un pull request, en GitHub, y un merge request, en GitLab, cumplen justamente esa función de propuesta. Son solicitudes para que cambios hechos en una rama, o en un fork, se integren en otra rama. Pero no son simplemente un botón para unir código. Son también un espacio de conversación. Ahí se revisa el código, se comentan decisiones, se detectan mejoras, se aprueba o se pide ajuste. Es una instancia técnica, sí, pero también muy cultural. Habla de cómo un equipo conversa sobre calidad, claridad y responsabilidad compartida.

Ese punto es precioso, porque nos muestra que colaborar no es solamente “subir cambios”. Colaborar es hacer visibles las decisiones, aceptar revisión, explicar por qué se hizo algo y construir confianza técnica entre personas. En un buen flujo de trabajo, nadie mete cambios importantes a escondidas. Se proponen, se miran, se discuten y después se integran.

La clase también conecta esto con los issues, que son una manera de registrar tareas, problemas, mejoras o ideas pendientes. Un issue, dicho de forma sencilla, es una conversación estructurada sobre algo que hay que resolver. Puede ser un bug, una nueva funcionalidad, una duda o una deuda técnica. Lo interesante es que, cuando el flujo está bien armado, el issue no queda aislado. Se conecta con ramas, commits y pull requests. Entonces el proyecto empieza a contar una historia completa: apareció un problema, se discutió, alguien trabajó en una solución, propuso cambios, se revisó y finalmente se integró.

Eso vuelve al desarrollo mucho más profesional. Ya no se trata de “alguien tocó algo y ahora funciona distinto”. Se trata de tener trazabilidad. Trazabilidad significa poder seguir el rastro de una decisión. Saber de dónde salió, quién la implementó, qué discusión hubo alrededor y en qué momento entró al proyecto. En equipos grandes, esa capacidad vale oro.

También es importante hablar de la instalación y configuración inicial de Git, porque aunque parezca una parte menor, tiene mucho sentido. Configurar el nombre y el correo no es un simple formalismo. Es decirle a la historia del proyecto quién sos vos. Cuando un commit queda registrado, también queda asociada la identidad de quien lo hizo. Eso forma parte de la responsabilidad y de la transparencia del trabajo colaborativo.

Y si uno se detiene un poco más, empieza a ver algo interesante. Git no solo sirve para guardar archivos. Sirve para pensar mejor. Obliga a agrupar cambios con sentido. Obliga a escribir mensajes claros. Obliga a decidir cuándo una modificación está lista para entrar en la historia. Obliga a mirar lo que otras personas hicieron y a integrar sin destruir. Es decir, además de ser una herramienta técnica, también educa una forma de trabajar más ordenada.

Por eso, cuando alguien empieza con Git, a veces siente que está aprendiendo comandos. Pero en realidad está aprendiendo una disciplina. Una forma de registrar trabajo, dividir problemas, coordinar esfuerzos y construir software con memoria. Y cuando eso se combina con GitHub o GitLab, aparece algo todavía más potente: una práctica colaborativa visible, revisable y sostenible.

Si quisiéramos resumir esta clase con una imagen simple, podríamos decir que Git es la memoria viva del proyecto. GitHub y GitLab son las plazas donde esa memoria se comparte, se discute y se enriquece con otras personas. Las ramas son caminos paralelos donde se exploran ideas. Los commits son hitos del recorrido. Los pull requests y merge requests son puertas de integración. Y los issues son recordatorios de lo que todavía hay que pensar, arreglar o mejorar.

Cuando todo eso encaja, el desarrollo deja de parecer una acumulación caótica de archivos y empieza a verse como un proceso con historia, conversación y dirección. Y ese cambio, aunque parezca silencioso, es enorme. Porque a partir de ahí ya no solo escribimos código... empezamos a construir proyectos de una forma mucho más consciente, mucho más colaborativa y mucho más profesional.
